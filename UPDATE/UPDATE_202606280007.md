## 3. Architectural Deep-Dive

### 3.1 Remote Command & Control (C2) Interface
The system has migrated from a unidirectional push-alert script to a bi-directional polling application. 
* A 10-second asynchronous offset listener queries the Telegram API without blocking the 0.5s execution loop.
* Variables such as `GATEKEEPER_THRESHOLD`, `STRATEGY_TYPE`, and risk management parameters are shifted to the global scope, allowing instant, real-time modification of the system's trading logic via phone.

### 3.2 Dynamic Reversal Execution
The `execute_trade()` function now behaves as an active position manager. 
* If the `ALLOW_MULTIPLE_TRADES` lock is active, the script checks the direction of the newly generated AI signal against the currently open MT5 ticket.
* If an opposite signal clears the TCN Gatekeeper, the script compiles a specific counter-order to immediately close the existing ticket, effectively capping drawdown before executing the new trend direction. 

### 3.3 Weekend Gap Protection (State Machine)
To protect margin from catastrophic weekend geopolitical gaps, the system tracks the broker's active tick timestamps.
* **Friday Liquidation:** At exactly 23:50 MT5 Server Time on Friday, `liquidate_weekend_positions()` triggers. It compiles FOK counter-orders for all active tickets, force-closes them, and switches `IS_BOT_ACTIVE` to `False`.
* **Deep Sleep:** The heavy 400-candle Pandas/ONNX inference block is bypassed, saving massive CPU cycles over the 48-hour weekend.
* **Monday Auto-Wake:** The system detects the first `weekday() == 0` tick at market open, flips `IS_BOT_ACTIVE` to `True`, and immediately resumes quantitative analysis.

### 3.4 Telemetry & Audit Precision
* **Closure Reason Tracking:** The `check_closed_trades()` module now parses internal MT5 Deal Reason Codes (e.g., `DEAL_REASON_TP`, `DEAL_REASON_SL`) and broker string comments to provide exact attribution for why a trade exited the market.
* **Skipped Trade Analytics:** Math calculations (Entry, SL, TP) were refactored to occur *before* the Execution Lock check. This ensures the `.log` text file permanently records the theoretical price targets of missed trades for future strategy evaluation.
